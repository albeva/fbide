# FBIde 0.5 — Pre-Release Code Audit

**Date:** 2026-06-02 · **Scope:** all of `src/lib` (~34 kLOC, 16 subsystems) · **Target:** 0.5.0 FINAL

Categories audited: memory leaks · race conditions · performance · code smells · bad practices.
Evaluated against the project standards (`CLAUDE.local.md`), modern C++23, and wxWidgets idioms.

> **Method:** 17 finders (13 per-subsystem + 4 cross-cutting: concurrency, ownership, performance, standards)
> produced 61 raw findings; **every** finding was then adversarially re-verified against the actual code.
> 56 survived (54 confirmed, 2 likely); 5 were dismissed as false positives (listed at the end so they
> aren't re-investigated). Severities below are the *post-verification* ratings, with live-vs-latent noted honestly.

## Critical

### C1 · Shared keyword tables are mutated under the intellisense worker's feet · 🚫 Ignored
`src/lib/editor/lexilla/FBSciLexer.cpp:49-51, 116-123, 665-666, 772` — race-condition

> **Won't fix:** the concurrent writer/reader window described below cannot actually occur in practice; marked ignored.
*Corroborated independently by 5 finder agents (config, editor, lexer, ownership, standards).*

The three keyword maps are **process-global mutable state**:

```cpp
KeywordMap kFreeBasic;     // FBSciLexer.cpp:49
KeywordMap kPreprocessor;  // :50
KeywordMap kAssembly;      // :51   (std::unordered_map)
```

- **Writer (UI thread):** `FBSciLexer::setKeywords` (`:116-123`) does `KeywordTable::clear()` then `setWords()` → `map.emplace(...)`, i.e. it **erases and rebuilds the live maps in place**. Reached on every Settings Apply/OK via `SettingsDialog::applyChanges` (`SettingsDialog.cpp:146`) → `UIManager::updateSettings` (`UIManager.cpp:698-703`) → `lexer::setFbKeywords` (`StyleLexer.cpp:23`); also from `ThemePage.cpp:349` and `DocumentManager.cpp:468`.
- **Reader (worker thread):** `IntellisenseService::process` calls `m_lexer->Lex(...)` (`IntellisenseService.cpp:126`), which reads the same globals — `map.find(...)` in `identifyKeyword` (`:665-666`) and `kPreprocessor.contains(...)` (`:772`). The worker holds **no lock** during `Lex` — `Entry()` releases `m_mtx` at `IntellisenseService.cpp:117` before calling `process()` at `:120`; `m_mtx` only guards `m_pending`/`m_stopRequested`.

Concurrent `clear()`/`emplace()` vs `find()`/`contains()` on a `std::unordered_map` is a data race = **undefined behaviour**: torn bucket reads, use-after-free of rehash-relocated nodes, or a hard crash on the worker thread.

The header comment at `FBSciLexer.hpp:52-54` explicitly promises *"The new table is published atomically … safe to call from the UI thread while a worker-thread lexer runs"* — **no such atomic publication exists.** (The startup call at `App.cpp:293` is *not* a race participant — it runs before the worker thread exists.)

- **Trigger:** edit a FreeBASIC file (a throttled parse is in flight or pending), open Settings → Keywords/Theme, click Apply. The dialog is modal on the UI thread only; the worker keeps lexing. Narrow window, but the consequence is heap corruption.
- **Fix:** make publication actually atomic, as the comment already claims. Hold the three maps in one `struct Tables` behind `std::atomic<std::shared_ptr<const Tables>>`; `setKeywords` builds a fresh `Tables` and `store`s it; `Lex` pins one `load()`ed snapshot at entry and reads through it for the whole call. (Alternative: a `std::shared_mutex` — `shared_lock` in `Lex`, `unique_lock` in `setKeywords` — or route the swap through the worker between tasks.) The atomic-snapshot approach is lock-free on the hot read path and matches the documented design.

## High

### H1 · Compiler process outlives its `BuildTask` → leak + use-after-free on shutdown · ✅ Fixed
`src/lib/compiler/BuildTask.cpp` · `AsyncProcess.cpp` — memory-leak / race

> **Resolved (all three defects):** (1) `UIManager::onClose` now calls `CompilerManager::killProcess()` before teardown, and `~BuildTask` calls `AsyncProcess::detach()` (clears the captured callback) so a late `OnTerminate` can't touch the freed task. (2) `m_running`/`setCompilerState(Compiling)` are now set *before* the synchronous version probe, so a re-entrant build dispatched inside the probe's nested loop hits the active-document guard. (3) `AsyncProcess::exec` returns `nullptr` on launch failure instead of a freed `self`. The false `CompilerManager.hpp` "replacing m_task aborts any process" comment was corrected.
*(Corroborated by compiler, xc-concurrency, xc-memory, xc-standards. Likely the #104 "crash after compiling" family.)*

`BuildTask` owns its child process only through a raw, self-deleting pointer and has **no destructor**:

```cpp
AsyncProcess* m_process = nullptr;   // BuildTask.hpp:98 ; class is NO_COPY_AND_MOVE, no dtor
```

`AsyncProcess` self-deletes in `OnTerminate` (`AsyncProcess.cpp:61`, `delete this`), and every callback handed to it **captures the `BuildTask` by reference**:

```cpp
m_process = AsyncProcess::exec(cmdStr, m_buildDir, true, [&](const ProcessResult& result){
    m_process = nullptr; ... m_ctx.getUIManager()...; onCompileFinished(result);   // BuildTask.cpp:102-108 (and run() :62-67)
});
```

There are **three** distinct lifetime defects here:

1. **Shutdown UAF (the live one).** `UIManager::onClose` (`UIManager.cpp:56`) has no in-flight-build check and never kills the process; `prepareToQuit` only handles unsaved docs. `~Context` → `~CompilerManager` (`=default`) → `~BuildTask` then frees the task while the child is still running. The process is **leaked + left detached** (`wxProcess::Detach()` is never called); and if `OnTerminate` is dispatched during the shutdown event-drain, the `[&]` callback dereferences the freed `BuildTask` and `m_ctx` → use-after-free. Quitting mid-compile is easy on Linux/macOS where `fbc` drives `gcc`/`ld` for seconds.
   *Minimal fix:* `~BuildTask() { if (m_process != nullptr) m_process->Detach(); }` (never `delete` — the process may still run), **and** have `UIManager::onClose` kill an in-flight build before teardown.

2. **Re-entrant replacement during the version probe** (`BuildTask.cpp:74-108`, Medium on its own). `startCompiler` runs the *synchronous* `probeCompilerVersion` (`:97`, a nested `wxExecute` event loop) **before** `m_running = true` (`:99`) and before `setCompilerState(Compiling)` (`:100`). The `getActiveDocument()` re-entry guard (`CompilerManager.cpp:303`) keys on `isRunning()` (== `m_running`, still `false`), so a Compile command dispatched inside the nested loop runs `m_task = make_unique<BuildTask>(...)`, **freeing the `BuildTask` whose `startCompiler` is on the stack**. On unwind, lines `:99-108` write through freed memory.
   *Fix:* set `m_running = true` and `setCompilerState(Compiling)` **before** the probe.

3. **Dangling `m_process` after a failed launch** (Low, latent). When `wxExecute` returns `pid == 0`, `AsyncProcess::exec` runs the callback and `delete this` (`AsyncProcess.cpp:33-36`), then the static factory `return self;` (`:13`) hands back a **freed pointer**, which `m_process = exec(...)` stores — clobbering the `nullptr` the callback just wrote. Not dereferenced today (`kill()` is guarded by `isRunning()`, and the callback set `m_running = false`), but the invariant "`m_process != nullptr` ⇒ a process exists" is broken.
   *Fix:* don't store the factory return; let the callback be the sole writer of `m_process` (or return `nullptr` on launch failure).

> Also: the invariant documented at `CompilerManager.hpp:28-30` ("replacing `m_task` aborts any process it spawned") is **false** — correct the comment too.

### H2 · Theme dropdown index desync → heap out-of-bounds read
`src/lib/settings/panels/ThemePage.cpp:312-318, 379-382` — correctness

`createTopRow()` builds the dropdown as `["New theme…", file₀ … fileₙ₋₁]` and stores the *n* paths in `m_themeFiles`. `onSelectTheme()` maps selection→path with **no bounds check**: `m_activeTheme = m_themeFiles[index - FILE_OFFSET];` (`:315`, `FILE_OFFSET == 1`). But `saveNewTheme()` appends to the dropdown **without** pushing the new path onto `m_themeFiles`:

```cpp
m_themeChoice->Append(name); m_themeChoice->SetStringSelection(name);   // :380-381 — m_themeFiles NOT updated
```

After a Save, re-selecting the new theme computes `m_themeFiles[n]` — one past the end (valid: `0..n-1`) — copying a garbage `wxString` into `m_activeTheme`, then into `m_theme.load()`. **UB; can crash or load a bogus path.** Reachable in a normal session: create a theme via Save, pick another, pick the new one again.

*Fix:* push the saved path onto `m_themeFiles` in lockstep with `Append(name)`; add a bounds check in `onSelectTheme`.

### H3 · Locale-change restart fires even when a later panel's validation fails · ✅ Fixed
`src/lib/settings/panels/GeneralPage.cpp:204-227` · `SettingsDialog.cpp:119-143` — correctness

> **Resolved:** `SettingsDialog::applyChanges` now runs every panel's `validate()` (active tab first) before any panel's `apply()` commits, and `Panel::apply()` is `void` so the language-restart prompt only fires from the post-validation commit pass. A Compiler-tab validation failure now keeps the dialog open with nothing applied.

`GeneralPage::apply()` (contract: "always succeeds") shows the language-restart prompt and, on **Yes**, queues `scheduleRestart(...)` via `CallAfter` (`:221-226`). `applyChanges()` applies the active page first, then the rest in `{General, Theme, Keywords, Compiler}` order — and only `CompilerPage::apply()` can return `false`. So if the user changes the language *and* has an empty Compiler config name, `applyOne(General)` queues the restart and returns `true`, then `applyOne(Compiler)` returns `false`; `applyChanges()` returns `false` and `EndModal` is skipped — **but nothing cancels the queued `CallAfter`**, which fires on the next tick and tears the IDE down (session saved, all docs closed, relaunch, `Exit()`). The dialog was supposed to stay open so the user could fix the name.

*Fix:* split apply into a **validate-all** pass and a **commit-all** pass; schedule irreversible work (the relaunch) only after every panel validates. (Or run Compiler's empty-name check up front.)

### H4 · Multi-line edits re-lex the whole document prefix · ✅ Fixed
`src/lib/editor/CodeTransformer.cpp:260-267` — performance

> **Resolved:** `transformRange` now colourises from `min(GetEndStyled(), PositionFromLine(LineFromPosition(rangeStart)))` instead of `0`, reusing already-styled prefix and the enclosing-line `LineState`.

`transformRange` starts with `editor.Colourise(0, rangeEnd);` (`:262`) — passing `start = 0` forces Scintilla to re-lex the **entire prefix** of the file up to `rangeEnd`, before tokenising only `[rangeStart, rangeEnd]`. This is on the editor hot path: `Editor::onModified` → `flushPendingInsert` → `onTextInserted` → `transformRange` (`Editor.cpp:999-1027`, `CodeTransformer.cpp:102`). So **every paste, drag-drop, or multi-line auto-indent** re-lexes from the top — O(prefix length), worst when editing near the end of a large file (this is the rc.1 "perf issues with multiline edits" area).

*Fix:* `editor.Colourise(rangeStart, rangeEnd)` — Scintilla rounds the start down to the enclosing line and reuses the persisted prior-line `LineState`, so styles through `rangeEnd` stay valid for the subsequent tokenise.

## Verified non-issues (investigated, dismissed)

These were raised and then refuted against the code — recorded so they aren't re-investigated.

- **AsyncProcess pipe-buffer deadlock** (`AsyncProcess.cpp:55-62`) — premise wrong. wx's async exec drains the redirected pipes via the event loop as data arrives; `OnTerminate` reads the *remainder*. No deadlock from large compiler output.
- **Symbol-browser hash collision skipping a rebuild** (`SymbolBrowser.cpp:290-303`) — `hashVector` is order-sensitive (non-commutative `hashCombine`, `SymbolTable.cpp:25-32`), so two documents with the same symbol *set* but different order do **not** collide; no stale tree wired to a new table.
- **BuildTask UAF via mid-build `m_task` replacement (normal commands)** — genuinely blocked: `getActiveDocument()` returns `nullptr` while `m_task->isRunning()` (`CompilerManager.cpp:302-305`). *(The real lifetime defects are via shutdown and the pre-`m_running` probe loop — see H1; those survive.)*
- **CompilerManager `m_task` reassignment race** (`CompilerManager.cpp:46/61/89/121`) — same `isRunning()` guard blocks it.
- **`findIncludeAt` linear scan** (`SymbolTable.cpp:270-277`) — real linear scan, but only called on a deliberate Ctrl+click hotspot gesture (`Editor.cpp:952`), not a hot loop. Negligible.

## Appendix — themes for 0.5.x

- **Two concurrency surfaces, both with sharp edges.** The intellisense worker (C1, M9) and the async compiler (H1) are the only threads in an otherwise single-threaded app, and both share state with the UI thread through mechanisms that are *documented as safe but aren't*. The cheapest durable fix for C1 is to implement the atomic publication the header already promises; for H1, give `BuildTask` a destructor that `Detach()`es and have shutdown kill in-flight builds.
- **"Validate then commit" recurs** (H3, M3): the Settings dialog performs irreversible side effects (disk writes, live state mutation, an app relaunch) during the per-panel apply loop, before all panels have validated. A single validate-all/commit-all split fixes both.
- **Range-honesty in the lexer** (H4, M8, plus Lows): several lexer/transform paths do whole-document work for a sub-range request (relex from 0, reserve from full length, over-read past the range end). Threading the requested range through consistently removes a class of per-keystroke cost on large files.
